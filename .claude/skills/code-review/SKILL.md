# 代码审查 Skill

对代码进行专业、深度的审查，发现潜在问题并提供改进建议。

## 审查原则

遵循以下核心原则进行审查：

1. **🎯 适度原则**：只建议必要的改动，不要过度设计
2. **🔍 证据驱动**：提出问题前确认真的存在该问题
3. **📐 简单优先**：优先使用简单的解决方案，除非有充分理由
4. **🚫 避免教条**：不要机械套用"最佳实践"，要结合实际场景判断

## 参数说明

- `$ARGUMENTS`: 可选的审查范围选项
  - `1`: 审查已推送的最近一次 commit
  - `2`: 审查未推送的所有 commits
  - `3`: 审查所有待定的文件
  - `4`: 审查指定文件
  - 未提供或其他值：询问用户选择

## 执行步骤

### 1. 确认审查范围

**检查 `$ARGUMENTS` 参数**：
- 如果 `$ARGUMENTS` 为 `1`、`2`、`3` 或 `4`：直接执行对应选项，跳过询问步骤
- 如果 `$ARGUMENTS` 为空或其他值：使用 AskUserQuestion 询问用户

**询问用户审查范围**（当无有效参数时，使用 AskUserQuestion）：

```json
{
  "question": "请选择代码审查范围：",
  "header": "审查范围",
  "multiSelect": false,
  "options": [
    {
      "label": "最近一次 commit",
      "description": "审查已推送到远程的最近一次 commit"
    },
    {
      "label": "未推送的 commits",
      "description": "审查当前分支所有未推送到远程的 commits"
    },
    {
      "label": "所有待定文件",
      "description": "审查所有已修改但未提交的文件"
    },
    {
      "label": "指定文件",
      "description": "选择或输入要审查的具体文件"
    }
  ]
}
```

**根据参数或用户选择执行相应操作**：

```bash
# 选项 1: 已推送的最近一次 commit
git diff @{u}~1..@{u}

# 选项 2: 未推送的所有 commits
git diff @{u}..HEAD

# 选项 3: 所有待定的文件
git diff --name-only HEAD
# 读取所有待定文件的完整内容

# 选项 4: 指定文件（用户提供文件路径）
# 使用 AskUserQuestion 展示已修改的文件列表
# 或直接使用 Read 工具读取用户指定的文件
```

### 2. 读取代码

对于每个文件：
1. 使用 **Read** 工具读取完整文件内容（理解上下文）
2. 使用 **Grep** 查找相关的依赖、调用关系

### 3. 深度分析

按照以下 checklist 进行审查：

#### 🔴 必查项（高优先级）

##### 线程安全
- [ ] 是否存在**真正的**多线程并发访问？
- [ ] 图片加载回调（Glide/Picasso）是否误用了原子类？
- [ ] UI 回调（onClick、postDelayed、LiveData.observe）是否误用了原子类？
- [ ] 后台线程 + 主线程访问共享变量是否使用了同步机制？

**判断标准**：
- ✅ 需要原子类：后台线程写入 + 主线程读取
- ❌ 不需要原子类：所有操作都在主线程（UI 回调、图片加载回调等）

##### 内存泄漏
- [ ] 全局单例是否持有 Activity/Fragment/Dialog/Context？
- [ ] 是否使用了 WeakReference？
- [ ] Dialog/Activity 生命周期管理是否正确？

**判断标准**：
- ✅ 需要弱引用：全局单例（object/static）持有短生命周期对象
- ❌ 不需要弱引用：Activity/Fragment 成员变量持有 Dialog

##### 空值安全
- [ ] 可空类型是否使用了 **?.** 或明确的空值检查？
- [ ] 是否有 **!!** 强制解包（需要有充分理由）？
- [ ] 是否检查了 "null" 字符串（数据序列化问题）？

##### 生命周期
- [ ] Fragment 操作前是否检查了 **isAdded/isResumed**？
- [ ] 异步回调中是否检查了生命周期状态？
- [ ] **onDestroyView()** 是否清理了所有监听器和资源？

#### ⚠️ 中频问题

- [ ] RecyclerView position 是否检查了 **< 0** 的情况？
- [ ] RecyclerView **onBindViewHolder()** 开头是否重置了所有状态？
- [ ] debounce 延迟时间是否合理（消息类 ≤1500ms）？
- [ ] **Observable.just()** 是否正确处理线程调度？

## 避免过度设计

### 常见误判场景

#### 1. 原子类误用

**❌ 错误判断**：看到计数器就建议用 AtomicInteger
**✅ 正确判断**：先确认是否存在多线程竞争

**检查方法**：
1. 追踪变量的所有读写位置
2. 确认是否有跨线程访问
3. 特别注意：Glide、UI 回调、postDelayed 都在主线程

#### 2. 弱引用误用

**❌ 错误判断**：看到单例持有引用就建议改成弱引用
**✅ 正确判断**：判断持有者和被持有对象的生命周期关系

**口诀**：
> "全局单例持有短生命周期对象 → 必须用弱引用"

#### 3. 同步锁误用

**❌ 错误判断**：看到共享变量就建议加锁
**✅ 正确判断**：确认是否在多个线程中访问

## 参考标准

### 线程安全判断表

| 场景 | 是否需要线程安全 | 原因 |
|------|----------------|------|
| Glide/Picasso 回调 | ❌ 否 | 主线程回调 |
| View.postDelayed() | ❌ 否 | 主线程消息队列 |
| LiveData.observe() | ❌ 否 | 主线程 Observer |
| 后台线程 + 主线程 | ✅ 是 | 真正的竞争 |

### 弱引用判断表

| 持有者 | 被持有对象 | 是否需要弱引用 |
|--------|-----------|---------------|
| 全局单例 | Activity/Dialog | ✅ 必须 |
| Activity 成员 | Dialog | ❌ 否 |

## 注意事项

1. **不要审查格式问题**：缩进、空格、import 顺序等由格式化工具处理
2. **提供充分理由**：每个问题都要说明"为什么"和"依据"
3. **区分优先级**：关键问题 vs. 优化建议
4. **给出具体方案**：不只是指出问题，要给出修复代码
5. **引用权威资料**：《Effective Java》、《Java Concurrency in Practice》、Android 文档

## 示例场景

### 场景 1: 用户说"review 这段代码"

1. 使用 AskUserQuestion 询问是否是最近的 commit
2. 执行 `git diff HEAD~1..HEAD`
3. 读取改动的文件完整内容
4. 按照 checklist 逐项检查
5. 输出审查报告

### 场景 2: 用户提供文件路径

1. 使用 Read 读取文件
2. 使用 Grep 查找相关调用
3. 深度分析
4. 输出审查报告

### 场景 3: Review 未推送的 commits

1. 执行 `git diff origin/develop..HEAD`
2. 读取所有改动的文件
3. 按项目规则审查
4. 输出审查报告
