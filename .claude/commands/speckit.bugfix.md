## 用户输入

```text
$ARGUMENTS
```

在继续之前, 你**必须**考虑用户输入. 用户输入可能包含 PM 工单号/链接、问题描述、或两者的组合.

## 目标

针对已有功能中的 Bug 进行系统化的诊断和修复:
1. 结合 PM 工单信息和用户描述, 深度理解问题的本质
2. 在代码和设计文档中定位问题根因, 判断是实现错误还是设计缺陷
3. 输出清晰的错误原因分析和修改方案供用户确认
4. 用户确认后更新相关设计文档, 生成修复任务并实施

## 执行步骤

### 步骤 1: 初始化上下文

从仓库根目录运行一次 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only` 并解析 JSON 以获取 FEATURE_DIR. 推导绝对路径:

- SPEC = FEATURE_DIR/spec.md (只读基准)
- PLAN = FEATURE_DIR/plan.md (可修改)
- DATA_MODEL = FEATURE_DIR/data-model.md (可修改)
- RESEARCH = FEATURE_DIR/research.md (可修改)
- TASKS = FEATURE_DIR/tasks.md (后续更新)

如果脚本未找到或执行失败, 回退到检测规范目录:
- 在仓库中搜索 `specs/*/spec.md` 和 `specs/*/*/spec.md` 模式
- 使用与当前分支名匹配或最近修改的目录

对于参数中的单引号如 "I'm Groot", 使用转义语法: 例如 'I'\''m Groot'(或尽可能使用双引号: "I'm Groot").

### 步骤 2: 收集问题信息

解析 $ARGUMENTS, 识别以下内容:

#### 2.1 PM 工单信息

**如果用户提供了 PM 工单号或链接**:

启动一个 Agent 子进程获取 PM 工单详情:
- 使用 Skill 工具调用 `pm-netease:pm-skill` 获取工单信息
- 提取: 工单标题、描述、优先级、状态
- 如果有子单, 获取相关子单信息

**如果用户未提供 PM 工单**:
- 仅使用用户描述作为问题输入
- 可选: 询问用户是否有关联的 PM 工单(但不强制要求)

#### 2.2 用户自定义描述

从 $ARGUMENTS 中提取用户对问题的描述, 包括:
- 问题现象(实际行为 vs 期望行为)
- 复现步骤(如果提供)
- 影响范围(如果提供)
- 用户的初步判断(如果提供)

#### 2.3 信息整合

将 PM 工单信息和用户描述合并为统一的**问题定义**:

```
🐛 问题定义

**来源**: PM#{工单号} / 用户描述
**标题**: [问题标题]
**现象**: [实际行为描述]
**期望**: [期望行为描述]
**优先级**: [P1/P2/P3]
**补充信息**: [用户额外描述的内容]
```

### 步骤 3: 加载设计文档

按优先级加载相关文档:

**必需**:
- spec.md - 需求基准, 用于验证问题是否属于设计范围

**目标文档(可能需要修改)**:
- plan.md - 实施计划, 检查是否有设计错误
- data-model.md - 数据模型, 检查字段定义或关系是否有问题
- research.md - 技术决策, 检查技术选型是否导致问题

**可选文档**:
- quickstart.md - 验证指南
- checklists/ - 检查清单
- tasks.md - 现有任务列表

### 步骤 4: 深度诊断

使用 Agent 工具启动 Explore 代理, 执行系统化的问题诊断:

#### 4.1 代码层面诊断

```
诊断任务: 根据问题定义在代码库中定位问题根因

问题定义:
{步骤 2.3 的问题定义}

相关设计文档摘要:
- plan.md 关键设计点: {摘要}
- data-model.md 关键实体: {摘要}

调查方向:
1. 搜索与问题相关的代码实现
2. 分析代码逻辑是否与 plan.md 的设计一致
3. 检查数据模型的实际使用是否与 data-model.md 一致
4. 识别代码中的逻辑错误、边界条件遗漏或竞态条件
5. 检查是否存在与 spec.md 需求不符的实现

输出要求:
- 问题相关的代码位置(文件路径 + 行号 + 关键代码片段)
- 代码层面的问题分析
- 与设计文档的不一致之处(如有)
```

#### 4.2 设计层面诊断

基于代码诊断结果, 进一步分析设计文档:

1. **Plan 检查**: plan.md 中的设计方案是否正确覆盖了该场景?
2. **Data Model 检查**: data-model.md 中的字段定义、类型、关系是否支撑该功能?
3. **Research 检查**: research.md 中的技术决策是否导致了该问题?
4. **Spec 对照**: 该问题是否因为 spec.md 中的需求描述不够明确导致?

#### 4.3 问题分类

将问题归类为以下之一:

| 类型 | 说明 | 影响范围 |
|------|------|---------|
| **代码实现错误** | 实现与设计一致, 但代码逻辑有 Bug | 仅需修改代码 |
| **设计缺陷** | plan/data-model 设计不完整或有误 | 需更新设计文档 + 代码 |
| **需求遗漏** | spec 未覆盖的边界条件 | 需补充 spec + 设计 + 代码 |
| **技术决策失误** | research 中的技术选择导致问题 | 需更新 research + plan + 代码 |

### 步骤 5: 输出诊断结果和修改方案

向用户展示完整的诊断报告:

```markdown
## 🔍 Bug 诊断报告

### 问题概述
- **问题**: [一句话描述]
- **分类**: [代码实现错误 / 设计缺陷 / 需求遗漏 / 技术决策失误]
- **根因**: [问题的根本原因]

### 错误原因分析

**1. 代码层面**
- [文件路径:行号] [具体代码问题描述]
- [文件路径:行号] [具体代码问题描述]

**2. 设计层面** (如有)
- [plan.md 第X行] [设计问题描述]
- [data-model.md 第Y行] [数据模型问题描述]
- [research.md 第Z行] [技术决策问题描述]

**3. 影响范围**
- [列出受影响的功能模块/页面/场景]

### 修改方案

**文档修改** (如需要):

| 文件 | 修改类型 | 修改内容概述 |
|------|---------|-------------|
| plan.md | FIX | [具体修改描述] |
| data-model.md | FIX | [具体修改描述] |
| research.md | ADD | [具体修改描述] |

**代码修改**:

| 文件 | 修改类型 | 修改内容概述 |
|------|---------|-------------|
| [代码文件路径] | FIX | [具体修改描述] |
| [代码文件路径] | ADD | [具体修改描述] |

### 风险评估
- **修改风险**: [低/中/高]
- **回归风险**: [可能影响的其他功能]
- **建议验证**: [验证修复的方法]
```

向用户提供选项:
- **确认方案**: 确认修改方案, 继续执行修复
- **调整方案**: 修改方案后再确认
- **重新诊断**: 提供新的线索或方向, 重新执行诊断
- **取消**: 取消修复

等待用户响应:
- 如果用户选择"确认方案": 继续步骤 6
- 如果用户选择"调整方案": 询问调整内容, 修改方案, 重新展示
- 如果用户选择"重新诊断": 进入步骤 5.1
- 如果用户选择"取消": 结束执行

### 步骤 5.1: 重新诊断(当用户提供反馈时)

当用户对诊断结果不满意或有新的线索时:

1. **收集反馈**: 获取用户的意见或新线索
2. **重新理解问题**: 结合用户反馈更新问题定义
3. **重新探索代码**: 使用 Agent 工具启动 Explore 代理, 根据新方向重新调查
4. **重新分析文档**: 重新检查设计文档中是否有遗漏
5. **返回步骤 5**: 输出新的诊断报告和修改方案

**循环限制**: 最多执行 3 次重新诊断. 如果 3 次后仍未达成共识, 建议用户提供更多信息或换一种方式排查.

### 步骤 6: 更新设计文档

根据确认的修改方案, 按需更新设计文档.

**注意**: 如果问题分类为"代码实现错误"(设计正确, 仅代码有误), 可以跳过文档更新, 直接进入步骤 7.

#### 6.1 更新 plan.md (如需要)

修改可能包括:
- 修正实施步骤中的逻辑错误
- 补充遗漏的边界条件处理
- 添加 bugfix 相关的注释

**变更追踪格式:**
```markdown
<!-- BUGFIX: YYYY-MM-DD - PM#工单号 - [简要描述] -->
[修改后的内容]
```

#### 6.2 更新 data-model.md (如需要)

修改可能包括:
- 修正字段类型或默认值
- 添加遗漏的字段
- 修正实体关系

#### 6.3 更新 research.md (如需要)

修改可能包括:
- 记录 Bug 根因分析
- 更新技术决策
- 添加经验教训

**新条目格式:**
```markdown
### Bugfix: [标题] (PM#工单号)

**问题**: [问题描述]
**根因**: [根本原因]
**修复方案**: [选择的修复方式]
**经验教训**: [防止类似问题的建议]
**代码参考**: [相关代码位置]
```

### 步骤 7: 生成修复任务

文档更新完成后(或跳过文档更新后), 自动执行 `/speckit.tasks` 更新任务文档:

- 在现有 tasks.md 中以**增量模式**添加 bugfix 相关任务
- 新任务应清晰标注为 bugfix 类型
- 包含具体的文件路径和修改描述

**注意**: 此步骤通过 handoff 机制调用 `/speckit.tasks`, 传递以下上下文:
```
根据 bugfix 诊断结果更新任务列表.

Bug 描述: [问题定义]
修改方案: [确认的修改方案]
需要修改的文件: [文件列表]
```

### 步骤 8: 实施修复

任务更新完成后, 自动执行 `/speckit.implement` 实施修复代码.

## 行为规则

- **始终先理解问题** - 在提出方案前必须完成代码和文档的深度诊断
- **始终先读取 spec.md** - 确保理解功能的完整需求上下文
- **始终在修改前确认** - 展示诊断报告和修改方案, 等待用户批准
- **保留现有内容** - 更新文档时仅修改必要部分, 不重新格式化整个文件
- **保持一致性** - 遵循现有命名约定和格式
- **追踪变更** - 为 bugfix 相关变更添加标记(BUGFIX 注释)
- **先诊断再修复** - 必须先定位根因, 避免头痛医头的修补
- **验证影响范围** - 确保修复不会引入新问题
- **尊重用户判断** - 用户不认可方案时, 根据反馈重新诊断, 不强行推进
- **循环限制** - 重新诊断最多 3 次, 超过则建议用户提供更多信息

## 常见场景

### 场景 A: 纯代码 Bug (无设计问题)

用户: "PM#271500 列表滑动到底部时 crash"

诊断结果:
- 分类: 代码实现错误
- 根因: RecyclerView adapter 的 getItemCount() 返回了错误的值
- 设计文档无需修改

操作:
1. 收集 PM 工单信息
2. 定位 crash 相关代码
3. 确认 plan.md 设计正确, 问题在实现
4. 输出代码修改方案
5. 用户确认后跳过文档更新
6. 直接生成修复任务并实施

### 场景 B: 设计遗漏导致的 Bug

用户: "频控逻辑没有考虑账号切换的场景"

诊断结果:
- 分类: 设计缺陷
- 根因: plan.md 未考虑账号切换时的频控重置
- data-model.md 缺少用户维度的频控 key

操作:
1. 理解问题
2. 检查代码实现 - 确认按 plan 实现但 plan 不完整
3. 检查 plan.md - 发现遗漏
4. 检查 data-model.md - 发现缺少字段
5. 检查 research.md - 记录根因分析和经验教训
6. 输出设计修改 + 代码修改方案
7. 用户确认后更新 plan.md + data-model.md + research.md
8. 生成修复任务并实施

### 场景 C: 用户对方案不满意

用户: "PM#271600 视频播放黑屏"
第一次诊断: 认为是播放器初始化顺序问题
用户反馈: "不是初始化问题, 只在特定机型上出现"

操作:
1. 收集用户反馈(特定机型信息)
2. 重新探索代码, 关注硬件兼容性相关逻辑
3. 更新诊断结果
4. 重新展示方案
5. 用户确认后执行修复
