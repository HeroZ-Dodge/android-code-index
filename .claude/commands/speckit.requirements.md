## 用户输入

```text
$ARGUMENTS
```

在继续之前, 你**必须**考虑用户输入(如果不为空).

## 概述

目标: 收集并串联多个外部需求文档(PRD、UX 交互稿、UI 视觉稿、PM 工单), 理解完整的大需求, 按功能边界拆分为多个独立的 spec 点, 生成 `feature.md` 文档供后续 `/speckit.specify` 流程使用.

**此命令是 speckit 工作流的前置步骤**, 位于 specify 之前:

```
requirements → specify → clarify → plan → review → tasks → implement
```

## 执行流程

### 步骤 1: 解析用户输入并收集需求来源

从 `$ARGUMENTS` 中识别用户已提供的文档链接. 按以下规则分类:

| 类型 | 识别规则 |
|------|---------|
| PRD | POPO 文档链接(popo.netease.com)、其他文档链接、用户明确标注为 PRD 的内容 |
| UX 交互稿 | MasterGo 链接(mgp://mastergo.netease.com)、用户标注为 UX/交互的链接 |
| UI 视觉稿 | MasterGo 链接(用户标注为 UI/视觉)、蓝湖等设计工具链接 |
| PM 工单 | PM 易协作链接(pm.netease.com/issues/xxx、a19.pm.netease.com) |

### 步骤 2: 询问缺失的文档来源

对于用户**未提供**的文档类型, **逐个**向用户询问.

使用 AskUserQuestion 工具, 一次性询问所有缺失的来源:

```
请提供以下需求文档:

1. PM 工单链接(如 pm.netease.com 链接) **[必须]**
2. PRD 产品需求文档链接或路径 (可跳过)
3. UX 交互稿链接(如 MasterGo mgp:// 链接) (可跳过)
4. UI 视觉稿链接(如 MasterGo 链接) (可跳过)
```

**重要**: 如果用户已在 $ARGUMENTS 中提供了某类文档, 不要重复询问该类.

**PM 工单为必须项**: 如果用户未提供 PM 工单链接且拒绝补充, 提示"PM 工单为必须项, 请先创建 PM 工单后再执行此命令"并中止流程.

**其他文档可选**: PRD、UX、UI 文档类型允许跳过. 但至少需要 PM 工单才能继续.

### 步骤 3: 解析各来源文档内容

**⚠️ 上下文管理原则**: 为避免原始文档内容撑爆上下文导致后续步骤超时, **必须使用 Agent 子进程**获取和提取文档内容. Agent 返回的摘要信息进入主上下文, 原始文档内容留在子进程中.

**并行获取**: 对所有已收集的文档来源, 使用 Agent 工具**并行**启动子进程获取. 每个 Agent 负责一种文档类型:

#### 3.1 PRD 文档 (通过 Agent 获取)

启动一个 Agent 子进程, prompt 中指定:
- **POPO 文档链接**: 使用 Skill 工具调用 `popo-doc-exporter` 导出文档内容
- **其他在线文档**: 使用 WebFetch 获取内容
- **本地文件**: 使用 Read 工具读取

要求 Agent 提取并**仅返回以下摘要**:
- 业务背景和目标(2-3句)
- 功能模块划分(列表)
- 各功能点的简要描述(每项1-2句)
- 优先级信息(如有)
- 非功能性需求(如有)

#### 3.2 UX 交互稿 (通过 Agent 获取)

启动一个 Agent 子进程, prompt 中指定:
- **MasterGo 链接(mgp://)**: 使用 Skill 工具调用 `mg-skills:mastergo-dispatcher` 解析设计稿
- **其他链接**: 使用 WebFetch 获取

要求 Agent 提取并**仅返回以下摘要**:
- 交互流程和页面跳转逻辑
- 用户操作路径
- 状态变化(加载、空态、错误态等)
- 交互细节(动效、手势等)

#### 3.3 UI 视觉稿

- **仅记录链接**, 不获取不分析. UI 稿在 specify 阶段不影响需求定义, 仅在后续 plan 阶段作为视觉参考.
- **不需要启动 Agent**, 直接在主上下文中记录链接即可.

#### 3.4 PM 工单 (通过 Agent 获取)

启动一个 Agent 子进程, prompt 中指定:
- **PM 链接**: 使用 Skill 工具调用 `pm-netease:pm-skill` 获取工单详情
- 如果链接包含子单, 递归获取所有子单信息

要求 Agent 提取并**仅返回以下结构化数据**:
- **PM 父工单号**(如 267213)
- **PM 父工单标题**(如"漫剧整体升级一期-26年1月")
- 需求标题和描述(简要)
- **PM 子单列表**: 每个子单的 {子单号, 子标题, 描述(1-2句)}
- 优先级和状态
- 关联的人员信息

### 步骤 4: 需求理解与优先级排序

**⚠️ 核心原则: PRD 为唯一权威, UX 仅做辅助补充**

当提供了 PRD 文档时, **必须严格以 PRD 的描述为准**进行 spec 拆分:
- Spec 点的划分、命名、功能范围、优先级**完全依据 PRD 内容**
- UX 交互稿**仅用于补充** PRD 中未详细说明的交互细节(如状态变化、手势操作等), **不能**作为拆分 spec 点的依据
- **不要**将 UX 稿中出现但 PRD 未提及的功能点单独拆分为 spec 点
- **不要**让 UX 稿的页面结构影响 spec 点的划分逻辑

**理解优先级**(从高到低):

1. **PRD 文档** (最高优先级/唯一权威): 作为需求的**唯一权威来源**, 定义"做什么"和"为什么做", **spec 拆分必须完全基于 PRD 的功能模块和描述**
2. **UX 交互稿** (仅辅助补充): 仅用于补充 PRD 中交互细节不够清晰的部分, 不参与 spec 拆分决策
3. **PM 工单描述** (第三): 作为补充信息, 填补 PRD 和 UX 未覆盖的细节
4. **UI 视觉稿** (仅记录): 不参与需求理解, 仅记录链接供后续使用

**冲突解决**: 当不同来源存在冲突时:
- PRD 描述**绝对优先**于 UX 稿和 PM 工单
- UX 稿优先于 PM 工单描述
- 记录冲突点, 在 feature.md 的备注中标注

**整合分析**:
1. **严格以 PRD 的功能模块划分为骨架和唯一拆分依据**
2. 仅在 PRD 描述不够详细时, 用 UX 稿的交互细节做辅助补充
3. 用 PM 工单补充遗漏的信息
4. 形成对完整需求的统一理解

### 步骤 5: 拆分 Spec 点并生成结构化摘要

基于步骤 4 的理解, 将大需求拆分为多个独立的 spec 点. 遵循以下原则:

**拆分原则**:
- 每个 spec 点应是一个**可独立开发、测试、交付**的功能单元
- 每个 spec 点应有**明确的用户价值**
- 粒度适中: 不要太大(超过 5 个用户故事)也不要太小(不足以独立交付价值)
- 每个 spec 点之间的耦合度应尽量低

**命名规则**:
- SP-001, SP-002, SP-003... (按优先级排序)
- 每个 spec 点有一个简短名称(2-5 个中文词)

**为每个 spec 点生成**:
1. **功能概述**: 2-3 句话描述核心功能和用户价值
2. **PM 子单号**: 关联的 PM 子单号
3. **PM 子单标题**: 关联的 PM 子单标题
4. **需求来源**: 关联到具体的 PRD 章节、UX 页面、PM 工单
5. **specify 输入**: 一段可直接作为 `/speckit.specify` 参数的功能描述(中文, 简洁完整, 包含足够上下文)
6. **优先级**: P1/P2/P3, 基于 PRD 中的优先级信息和业务价值判断

**PM 子单关联规则**:
- 如果 PM 工单有子单 → 将 SP 点与子单一一映射
- 如果一个子单太大需要拆分为多个 SP → 使用同一个子单号, 在子标题中区分
- 如果 PM 工单无子单 → 由开发者在 PM 系统中创建对应子单,在文档中只需要拟定占位符标记

**⚠️ 关键: 输出结构化摘要并写入临时文件**

完成拆分后, 你**必须**:
1. 以纯文本形式输出一份精简的结构化摘要到对话中(供用户查看)
2. 同时使用 Write 工具将**完全相同的摘要内容**写入临时文件 `/tmp/speckit-summary.md`(供步骤 7 的 Agent 读取, 避免主上下文再次输出大量内容)

这一步至关重要 — 它将前面收集到的大量原始文档内容压缩为紧凑的结构化数据.

摘要格式:

```
=== 需求拆分摘要 ===
PM单号: {PM单号}
PM标题: {PM标题}
PM链接: {PM链接}
需求概览: {1-2句话的整体需求摘要}
来源: PRD={链接或"未提供"} | UX={链接或"未提供"} | UI={链接或"未提供"} | PM={链接}
本地文件: {逗号分隔的本地文件路径列表, 或"无"}

SP-001: {名称} | P{优先级} | 子单号={子单号} | 子标题={子标题}
  概述: {功能概述}
  来源: PRD={章节} UX={页面} PM={工单}
  specify: {specify 输入内容}

SP-002: ...
(继续列出所有 SP 点)

依赖: SP-001 → SP-003; SP-002 独立
执行顺序: 第一批[SP-001,SP-002] → 第二批[SP-003] → ...
=== 摘要结束 ===
```

**后续步骤 6、7 必须仅基于此摘要工作, 不要回溯引用前面的原始文档内容.**

### 步骤 6: 创建目录结构与分支

**检查分支并创建目录**: 执行 `git branch --show-current` 获取当前分支名, 然后调用脚本:
   - 如果当前分支名包含该 PM 单号(如 `feature/{PM单号}_xxx`) → 跳过分支创建:
     ```bash
     .specify/scripts/bash/create-new-feature.sh --json \
       --mode feature \
       --pm-id "{PM单号}" \
       --pm-title "{清理后的PM标题}" \
       --short-name "{从当前分支名提取的短名}" \
       --skip-branch \
       "{功能描述}"
     ```
   - 否则 → 生成英文短名(2-4词, 小写+连字符, 如 `manga-upgrade`)并创建分支:
     ```bash
     .specify/scripts/bash/create-new-feature.sh --json \
       --mode feature \
       --pm-id "{PM单号}" \
       --pm-title "{清理后的PM标题}" \
       --short-name "{英文短名}" \
       "{功能描述}"
     ```
   脚本返回 JSON 包含 BRANCH_NAME 和 FEATURE_DIR.

### 步骤 7: 生成 feature.md 并归档文档

**⚠️ 必须使用 Agent 子进程生成文件**, 避免主上下文输出大量文件内容导致 API 超时.

启动一个 Agent 子进程, prompt 中**只需传入 FEATURE_DIR 路径**(步骤 6 脚本返回的值), 不要在 prompt 中粘贴摘要内容.

要求 Agent 执行:
1. 读取临时文件 `/tmp/speckit-summary.md` 获取结构化摘要
2. 读取模板文件 `.specify/templates/feature-template.md`
3. 基于摘要 + 模板, 使用 Write 工具生成 `{FEATURE_DIR}/feature.md`
4. **归档本地文档**(如有): 如果摘要中记录了本地文件路径, 用**单条 bash 命令**完成:
   ```bash
   mkdir -p "{FEATURE_DIR}/docs" && cp "{文件1}" "{文件2}" ... "{FEATURE_DIR}/docs/"
   ```
   - 如果文件名冲突, 添加来源前缀: `prd_`, `ux_`, `ui_`, `pm_`
   - 在 feature.md 的元数据区域记录已归档的文档列表
5. 仅复制用户提供的**本地文件路径**, 不下载在线链接的内容
6. 删除临时文件: `rm -f /tmp/speckit-summary.md`
7. 完成后返回确认信息: 文件路径、SP 点数量

### 步骤 8: 报告完成 (直接输出, 不要额外读取文件)

输出以下信息:

```markdown
## 需求拆分完成

**目录路径**: specs/{PM单号}_{标题}/feature.md
**Feature 分支**: feature/{PM单号}_{英文短名}
**PM 工单**: #{PM单号} - {标题}
**需求来源**: [列出已解析的文档数量]
**拆分结果**: [N] 个 Spec 点

| ID | PM子单号 | 名称 | 优先级 | Spec分支 |
|----|---------|------|--------|---------|
| SP-001 | 267214 | 漫剧播放器升级 | P1 | feature/267213_267214_player-upgrade |
| SP-002 | ... | ... | ... | ... |

### 建议执行顺序
1. 第一批: SP-001, SP-002 (基础功能)
2. 第二批: SP-003 (依赖第一批)
...

### 下一步
对每个 spec 点运行 `/speckit.specify [specify 输入内容]`
建议从 SP-001 开始.
```

## 行为规则

- 如果用户未提供任何文档来源且拒绝补充, 提示至少需要一个来源并中止
- 如果某个 Skill 调用失败(如 MasterGo 链接无法解析), 记录错误并继续处理其他来源, 在备注中标注失败项
- 不要在此阶段深入到实现细节, 保持在"做什么"层面
- 每个 spec 点的 "specify 输入" 必须是自包含的描述, 不依赖 feature.md 的上下文即可理解
- UI 视觉稿仅记录链接, 不分析视觉细节, 不影响需求拆分
- 对于 PRD 中模糊或矛盾的需求, 在 spec 点的功能概述中标注 [待确认], 交由后续 specify/clarify 阶段处理
