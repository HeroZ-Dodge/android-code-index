## 用户输入

```text
$ARGUMENTS
```

在继续之前, 你**必须**考虑用户输入. 用户输入描述了计划文档中需要优化、增强或修复的内容.

## 目标

根据用户的输入内容, 对计划文档进行丰富、修正或补充. 这包括:
- 调研用户提到的技术点或实现细节
- 更新相关文档(plan.md、data-model.md、quickstart.md、research.md)
- 如果修改内容对应 review 文档中的问题, 标记为已处理

## 操作约束

**只读 spec.md**: 需求规格作为基准, 不修改

**可写计划文档**: 可修改 plan.md、data-model.md、quickstart.md、research.md

**更新审核状态**: 如果修改解决了 review 中的问题, 更新 `checklists/plan_review.md`

## 执行步骤

### 1. 初始化上下文

从仓库根目录运行一次 `.specify/scripts/bash/check-prerequisites.sh --json --paths-only` 并解析 JSON 以获取 FEATURE_DIR. 推导绝对路径:

- SPEC = FEATURE_DIR/spec.md (只读基准)
- PLAN = FEATURE_DIR/plan.md (可修改)
- DATA_MODEL = FEATURE_DIR/data-model.md (可修改)
- RESEARCH = FEATURE_DIR/research.md (可修改)
- QUICKSTART = FEATURE_DIR/quickstart.md (可修改)
- REVIEW = FEATURE_DIR/checklists/plan_review.md (状态更新)

如果脚本未找到或执行失败, 回退到检测规范目录:
- 在仓库中搜索 `specs/*/spec.md` 和 `specs/*/*/spec.md` 模式
- 使用与当前分支名匹配或最近修改的目录

对于参数中的单引号如 "I'm Groot", 使用转义语法: 例如 'I'\''m Groot'(或尽可能使用双引号: "I'm Groot").

### 2. 加载所有文档

加载所有可用文档以获取上下文:

**必需:**
- spec.md - 需求基准, 用于验证修改是否符合需求

**目标文档(将被修改):**
- plan.md - 实现计划
- data-model.md - 数据模型
- research.md - 研究分析
- quickstart.md - 快速开始/验证指南

**审核文档(用于状态更新):**
- checklists/plan_review.md - 如果存在, 读取以识别待处理问题

### 3. 分析用户请求并澄清模糊点

解析用户输入($ARGUMENTS)以理解:

1. **优化类型**:
   - `ADD` - 添加新内容(新字段、新步骤、新场景)
   - `FIX` - 修复问题(修正错误、补充遗漏)
   - `ENHANCE` - 增强细节(丰富描述、添加示例)
   - `CLARIFY` - 澄清模糊点(明确定义、消除歧义)

2. **目标范围**:
   - 涉及哪些文档?
   - 涉及哪些章节/实体/步骤?

3. **审核映射**:
   - 用户提到的内容是否对应 review 中的某个问题?
   - 记录问题 ID(如 "问题 1"、"CRITICAL_001" 等)

4. **模糊性检测**(关键):

   在继续之前, 分析用户输入中的模糊性或不清晰的点:

   **模糊性类别:**
   - **范围模糊**: 用户提到的范围不明确(如"优化这个功能"但未指明哪个功能)
   - **技术模糊**: 技术细节不清晰(如"添加缓存"但未指明缓存策略)
   - **边界模糊**: 边界条件不明确(如"处理异常情况"但未列举具体场景)
   - **优先级模糊**: 多个修改点但优先级不明(如"修复这些问题"但未说明顺序)
   - **数值模糊**: 具体数值或配置不明确(如"增加超时时间"但未指定多久)

   **如果检测到模糊性**, 进入**澄清循环**(步骤 3.1).

   **如果没有模糊性**, 直接继续步骤 4.

### 3.1 澄清循环(交互式)

当用户输入中检测到模糊性时:

1. **生成澄清问题**(每次会话最多 3 个问题):
   - 仅询问实质上影响优化结果的问题
   - 每个问题应可通过简短回答或多项选择来回答
   - 按以下优先级排序: (对实现的影响 * 模糊程度)

2. **一次只提出一个问题**:

   对于多项选择题:
   ```
   🤔 需要澄清:

   **问题**: [具体问题]

   **推荐**: 选项 [X] - [推荐理由]

   | 选项 | 描述 |
   |------|------|
   | A | <选项 A 描述> |
   | B | <选项 B 描述> |
   | C | <选项 C 描述> |
   | 自定义 | 提供其他答案 |

   回复选项字母, 或输入"推荐"采用建议选项, 或提供自定义答案.
   ```

   对于简答题:
   ```
   🤔 需要澄清:

   **问题**: [具体问题]

   **建议**: [你的建议答案] - [理由]

   回复"建议"采用推荐, 或提供你的答案(简短回答即可).
   ```

3. **处理用户响应**:
   - 如果用户回复"推荐"、"建议"、"yes"或"recommended": 使用你的建议答案
   - 如果用户提供有效答案: 记录它
   - 如果答案模糊: 要求澄清(不计为新问题)

4. **继续或前进**:
   - 每次回答后, 检查是否仍有模糊性
   - 在以下情况时停止提问:
     - 所有关键模糊性已解决
     - 用户发出完成信号("继续"、"开始"、"done"、"go ahead")
     - 达到最多 3 个问题
   - 记录所有澄清结果, 供后续步骤使用

5. **澄清摘要**:
   ```
   ✅ 澄清完成:

   - Q1: [问题] → A: [答案]
   - Q2: [问题] → A: [答案]

   现在开始调研和修改...
   ```

### 4. 调研与分析

根据用户请求, 执行必要的调研:

**技术细节方面:**
- 在代码库中搜索相关实现
- 分析现有代码模式
- 识别依赖和约束

**数据模型变更方面:**
- 检查现有实体定义
- 验证字段命名约定
- 确保类型一致性

**实施步骤方面:**
- 验证任务依赖关系
- 检查缺失的前置条件
- 验证执行顺序

**在修改前输出调研结果:**
```
📋 调研结果:
- [发现 1]
- [发现 2]
- ...

📝 计划修改:
- [文件]: [修改内容概述]
- ...
```

### 5. 与用户确认变更

在应用变更之前, 展示修改计划:

```
🔧 即将进行以下修改:

1. **plan.md**
   - [具体修改描述]

2. **data-model.md**
   - [具体修改描述]

3. **research.md**
   - [具体修改描述]

4. **quickstart.md**
   - [具体修改描述]

5. **checklists/plan_review.md** (状态更新)
   - 问题 X: 标记为已处理

请选择操作:
```

向用户提供选项:
- **确认**: 确认并执行修改
- **调整**: 调整修改方案
- **重新调研**: 重新调研后再修改
- **取消**: 取消操作

等待用户响应:
- 如果用户选择"确认": 继续步骤 6
- 如果用户选择"调整": 询问需要调整什么, 修改方案, 重新确认
- 如果用户选择"重新调研": 继续步骤 5.1(重新调研)
- 如果用户选择"取消": 结束执行

### 5.1 重新调研(可选)

如果用户请求重新调研, 执行更深入的代码调查:

**询问用户调研重点:**
```
请描述需要重新调研的内容:
例如: "调研现有的账号切换监听机制" 或 "查找 JSSDK 回调的实现方式"
```

**执行调研代理:**

使用 Task 工具启动 Explore 代理, 执行用户的调研请求:
```
调研任务: {用户输入的调研内容}

项目路径: {项目根目录}

调研要求:
1. 搜索相关代码实现
2. 分析现有模式和约定
3. 识别可复用的组件
4. 记录关键代码位置

输出格式:
- 发现的相关代码(文件路径 + 关键代码片段)
- 现有实现模式分析
- 建议的实现方案
- 需要注意的约束或依赖
```

**调研完成后:**
1. 向用户展示调研结果
2. 根据调研结果更新修改计划
3. 返回步骤 5 重新与用户确认

## 行为规则

- **始终先澄清模糊性** - 如果用户输入模糊或边界不清, 在继续之前先提出澄清问题
- **始终先读取 spec.md** - 确保变更与需求一致
- **始终在修改前确认** - 展示变更计划并等待批准
- **保留现有内容** - 仅修改必要部分, 不重新格式化整个文件
- **保持一致性** - 遵循现有命名约定和格式
- **追踪变更** - 为重大变更添加注释或标记
- **更新审核状态** - 将已解决的问题标记为已处理
- **具体明确** - 提供变更的确切位置和内容
- **先调研再修改** - 必要时调查代码库
- **验证变更** - 确保修改不与 spec.md 矛盾
- **限制澄清问题** - 最多问 3 个问题, 按影响优先级排序
- **提供推荐** - 提问时始终建议一个推荐选项

## 常见优化场景

### 场景 A: 添加缺失字段

用户: "data-model.md 缺少 tab4 的频控配置字段"

操作:
1. 读取 spec.md 验证 tab4 需求
2. 读取 data-model.md 当前结构
3. 调研现有字段命名模式
4. 添加 `userTab4MaxCount` 字段, 使用一致格式
5. 如需要, 更新 plan.md 中的相关方法
6. 将对应的审核问题标记为已解决

### 场景 B: 修复实现细节

用户: "plan.md 中账号切换的处理逻辑不完整"

操作:
1. 读取 spec.md 了解账号切换需求
2. 调研现有账号切换代码模式
3. 向 plan.md 添加详细实施步骤
4. 向 research.md 添加相关代码分析
5. 向 quickstart.md 添加验证步骤
6. 更新审核状态

### 场景 C: 增强代码示例

3. 更新 research.md 中的代码示例
4. 更新 plan.md 中的相关实施步骤
5. 将审核问题标记为已解决

### 场景 D: 澄清模糊定义

用户: "data-model.md 中 LandingActionType 的命名不一致"

操作:
1. 读取所有文档查找不一致之处
2. 确定正确的命名约定
3. 更新所有文档中的出现位置
4. 如需要, 添加术语说明
5. 更新审核状态

### 场景 E: 模糊的用户请求(需要澄清)

用户: "优化一下缓存逻辑"

```
🤔 需要澄清:

**问题**: 需要优化哪个模块的缓存逻辑?

| A | 首页数据缓存 |
| B | 用户数据缓存 |

| 自定义 | 指定其他模块 |

回复选项字母, 或输入"推荐"采用建议选项.
```

用户: "B"

```
🤔 需要澄清:

**问题**: 优化方向是什么?

**推荐**: 选项 A - 根据 review 中的性能问题反馈
```
