# Android Code Index 设计分享

> 一份通俗易懂的代码索引系统介绍

---

## 一、这个索引是干什么的？

### 1.1 用一句话描述

**Android Code Index 给你的代码项目建了一个"结构化索引"**，让你能像查字典一样查代码。

### 1.2 什么是"符号化"？

想象你有一个 100 万行代码的 Android 项目，传统搜索（如 grep、IDE 全局搜索）是怎么工作的？

```
传统搜索（文本匹配）:
  你搜 "FeedFragment"
  → 系统逐行扫描所有文件
  → 返回包含这个词的所有行
  → 结果混杂：类定义、变量声明、注释、import 语句全都有
```

而**符号化**是把代码"读懂"后再存储：

```
符号化索引（语义理解）:
  索引阶段就把代码解析成结构化的数据：

  {
    "name": "FeedFragment",
    "kind": "class",
    "parent_class": "BaseFragment",
    "interfaces": ["Parcelable"],
    "module": ":app",
    "file_path": "src/main/java/com/example/FeedFragment.kt",
    "line_number": 25,
    "methods": [
      {"name": "onCreateView", "return_type": "View"},
      {"name": "onDestroyView", "return_type": "Unit"}
    ]
  }

  你搜 "FeedFragment"
  → 直接返回这个类的结构化信息
  → 知道它是类、继承谁、实现什么接口、有哪些方法
```

**简单说**：符号化就是把"文本"变成"知识"。

### 1.3 什么是 FTS5？为什么快？

**FTS5** 是 SQLite 的全文搜索引擎，它快的原因是：

```
传统 grep 搜索:
  搜索 "Activity" → 遍历 10 万个文件 → 逐行匹配 → 耗时数秒到数十秒

FTS5 全文搜索:
  搜索 "Activity" → 查预建的倒排索引 → 直接返回匹配的行号 → 耗时几毫秒到几十毫秒
```

**倒排索引**是什么？就像书的目录：

```
普通搜索 = 从第一页读到最后一页找关键词
倒排索引 = 先查书后目录，直接翻到对应页码

FTS5 在索引阶段就建好了这样的"目录"：
  Activity → [文件 A 行 5, 文件 B 行 23, 文件 C 行 108, ...]
  Fragment → [文件 A 行 12, 文件 D 行 45, ...]

搜索时直接查这个"目录"，所以快。
```

**额外优化**：
- **BM25 评分**：搜索 "FeedFragment" 时，精确匹配的排前面，部分匹配的排后面
- **驼峰分词**：搜 "FeedFrag" 能匹配 "FeedFragment"（因为索引时拆分成了 "feed fragment"）
- **复合索引**：支持 "module=:app AND kind=class" 这样的组合过滤

## 二、系统定位：擅长什么，不擅长什么

### 2.1 擅长的事 ✅

| 场景 | 为什么擅长 | 性能对比 |
|------|-----------|---------|
| **找子类** | `get_subclasses("BaseFragment")` 直接查 `parent_class` 字段 | 3 秒 vs Grep 5 分钟 |
| **找接口实现** | `get_implementations("ICallback")` 查 `interfaces` JSON 字段 | 5 秒 vs Grep 3 分钟 |
| **找继承链** | `get_inheritance("MyClass")` 沿 `parent_class` 向上追溯 | 秒级返回完整链条 |
| **按元数据过滤** | "找所有返回 LiveData 的 public 方法" → 直接查 `return_type` + `visibility` 字段 | 2 秒 vs Grep 3 分钟 |
| **跨文件语义分析** | 索引已经建立了类与类之间的关系图 | Grep 完全做不到 |

### 2.2 不擅长的事 ❌

| 场景 | 为什么不擅长 | 建议方案 |
|------|-------------|---------|
| **搜索系统 SDK 类** | Android SDK（Fragment、Parcelable 等）没有被索引 | 用 Grep |
| **模糊关键词搜索** | "找高光视频相关代码" 这种语义模糊的需求 | 用 Grep 更直接 |
| **刚写的新代码** | 索引有延迟，需要等增量更新或手动 `update_index` | 先索引再查 |
| **第三方库代码** | 只索引项目源码，不索引依赖的 aar/jar | 用 IDE 反编译 |
| **配置文件内容** | build.gradle、XML 布局内容等 | 用 `read` 命令直接读文件 |

### 2.3 一句话总结

> **如果你经常需要分析继承体系、追踪接口实现、按方法签名搜索 → 索引系统是真神器**
>
> **如果你只是关键词搜索、查配置文件 → Grep/IDE 搜索更直接**

> MCP 工具的真正价值不在"更快更省"，而在"结构化"：
>

> 2. **继承/实现关系**：`get_subclasses`、`get_implementations` 是图查询，Grep 无法替代
> 3. **大文件导航**：`get_class_api` 快速浏览类结构，避免阅读全文
>

> - 核心挑战是理解代码语义，不是定位代码位置
> - 一旦定位到核心类，Read 全文读取 + 人工分析更直接
> - MCP 工具的精确符号定位优势发挥不出来

**推荐工具使用策略**：

| 任务阶段 | 推荐工具 | 理由 |
|------|---------|------|
| 1. 定位核心类 | MCP search_code | 快速定位，带行号 |
| 2. 浏览类结构 | MCP get_class_api | 方法列表一览 |
| 3. 理解业务逻辑 | Read 全文 | 上下文完整 |
| 4. 追踪调用链 | 混合使用 | get_symbol_source + Grep |
| 5. 分析继承关系 | MCP get_subclasses | 无可替代 |

> **最佳实践**：先用 MCP 工具定位，再用 Read 深入理解，而不是二选一。

## 四、系统架构（通俗版）

### 4.1 数据流向

```
源码文件 (.kt/.java/.xml)
     │
     ▼
 ┌─────────┐
 │ 解析器   │  ← 用 tree-sitter"读懂"代码
 │ Parsers │     提取类、方法、继承关系等
 └────┬────┘
      │
      ▼
 ┌─────────┐
 │ 索引器   │  ← 批量写入数据库
 │ Indexer │     全量索引 / 增量更新
 └────┬────┘
      │
      ▼
 ┌─────────────┐
 │   SQLite    │  ← 存符号 + 建索引
 │   Database  │     FTS5 全文搜索
 └──────┬──────┘
        │
        ▼
 ┌──────────────┐
 │  查询引擎     │  ← 封装所有查询逻辑
 │ QueryEngine  │     提供结构化 API
 └──────┬───────┘
        │
        ├──→ MCP Server (给 Claude Code 用)
        │
        └──→ HTTP API (给 Web UI 用)
```

### 4.2 核心组件

| 组件 | 职责 | 类比 |
|------|------|------|
| **解析器** | 读懂 Kotlin/Java/XML 代码，提取符号 | 像"图书编目员"，把书的内容分类整理 |
| **索引器** | 把符号写入数据库，支持增量更新 | 像"图书上架"，新书来了快速上架 |
| **数据库** | 存储符号和索引 | 像"图书馆的目录系统" |
| **查询引擎** | 提供查询方法 | 像"图书管理员"，帮你查书 |
| **MCP Server** | 给 AI 助手提供工具 | 像"专门为 AI 开的窗口" |

## 五、数据库里存了什么？

### 5.1 symbols 表（符号表）

每个符号一条记录，包含：

```
name            → 简单名：FeedFragment
qualified_name  → 全限定名：com.example.feed.FeedFragment
kind            → 类型：class/interface/function/property
parent_class    → 父类：BaseFragment
interfaces      → 实现的接口：["Parcelable", "Serializable"]
module          → 所属模块：:app
file_path       → 文件路径
line_number     → 定义行号
visibility      → 可见性：public/private
return_type     → 返回值类型（函数用）
src_code        → 源码（函数/构造函数用）
```

### 5.2 symbols_fts 表（全文搜索表）

这是 FTS5 虚拟表，用于快速搜索：

```
索引字段：name, qualified_name, annotations, name_tokens

特殊技能：
  • 前缀搜索：Activity → 匹配 Activity, ActivityManager
  • 驼峰分词：FeedFrag → 匹配 FeedFragment（因为索引时拆成 feed fragment）
  • 多词搜索：Base Fragment → 同时包含 Base 和 Fragment
```

### 5.3 关系是怎么存的？

```
继承关系 → parent_class 字段（存父类的 qualified_name）

接口实现 → interfaces 字段（JSON 数组存接口名）

模块依赖 → module_dependencies 表（记录 build.gradle 中的依赖）
```

## 六、索引是怎么建立的？

### 6.1 全量索引流程

```
1. 扫描项目
   找到所有 .kt / .java / .xml / .gradle 文件

2. 批量处理（每批 200 个文件）
   ┌──────────────────────────────────┐
   │ 对每个文件：                      │
   │   a. 清除旧记录（DELETE）        │
   │   b. 解析文件，提取符号          │
   │   c. 写入新记录（INSERT）        │
   │   d. 记录文件状态                │
   └──────────────────────────────────┘
   ↓
3. 提交事务（每 200 个文件提交一次）

4. 完成
   全量索引通常耗时 5-10 分钟（取决于项目规模）
```

### 6.2 增量更新流程

```
1. 对比文件修改时间
   已知数据库中的 last_modified
   对比当前文件的实际修改时间

2. 只处理变化的文件
   • 已删除 → DELETE 该文件的所有记录
   • 新增/修改 → 重新解析并更新
   • 未变化 → 跳过

3. 完成
   通常几秒内完成
```

### 6.3 文件监听（Watch 模式）

```
启动：python main.py index watch /path/to/project

原理：
  后台运行 watchdog 监听文件变化
  检测到变化后，等待 10 秒（debounce，避免频繁触发）
  然后自动执行增量更新

适用场景：
  写代码时保持运行，索引始终保持最新
```

## 七、查询 API 设计

### 7.1 三层查询能力

**P0 层：搜索定位（入口）**

```python
search_code(keyword, kind, module)      # 全文搜索
search_class(name, parent_class)        # 按条件找类
search_function(name, return_type)      # 按条件找函数
```

**P1 层：关系分析（进阶）**

```python
get_inheritance(class_name)             # 继承链
get_subclasses(class_name)              # 子类
get_implementations(interface_name)     # 接口实现
```

**P2 层：源码查看（深度）**

```python
get_class_api(class_name)               # 成员摘要（精简版）
get_class_api_full(class_name)          # 成员完整信息（含源码）
get_symbol_source(symbol_id)            # 单个符号源码
```

### 7.2 推荐工作流

```
用户问题："帮我找一下 FeedFragment 的继承关系"

步骤 1: search_code("FeedFragment", kind="class")
       → 拿到 qualified_name: com.example.feed.FeedFragment

步骤 2: get_inheritance("com.example.feed.FeedFragment")
       → 返回继承链

步骤 3: get_class_api(qualified_name)
       → 查看成员摘要

步骤 4: get_symbol_source(id)
       → 按需获取单个方法源码
```

## 八、为什么选择 SQLite？

### 8.1 其他选项的对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SQLite** | 零配置、单文件、FTS5 内置 | 并发写入受限 |
| Elasticsearch | 分布式、功能强大 | 部署复杂、内存消耗高 |
| PostgreSQL | 功能全面 | 需要单独部署服务 |

### 8.2 SQLite 够用的理由

```
• 单用户使用，并发不是问题
• FTS5 全文搜索性能足够（毫秒级）
• 单文件数据库，易于备份和迁移
• 无需额外部署，降低使用门槛
```

## 十、MCP 工具使用指南

### 10.1 工具分类

**搜索类** - 用于定位代码
- `search_code` - 全文搜索
- `search_class` - 找类
- `search_function` - 找函数
- `search_interface` - 找接口

**关系类** - 用于分析代码关系
- `get_inheritance` - 继承链
- `get_subclasses` - 子类
- `get_implementations` - 接口实现
- `get_class_interfaces` - 类实现的接口

**详情类** - 用于查看代码内容
- `get_file_symbols` - 文件内所有符号
- `get_class_api` - 类成员摘要
- `get_class_api_full` - 类成员完整信息
- `get_symbol_source` - 单个符号源码

**资源类** - 用于查找 Android 资源
- `find_layout` - 布局文件
- `find_drawable` - 图片资源
- `find_manifest_component` - 四大组件
- `find_style` - 样式资源

**工具类**
- `find_module_deps` - 模块依赖
- `project_stats` - 项目统计
- `update_index` - 增量更新索引

### 10.2 使用建议

```
✅ 优先用 MCP 工具的场景:
  • 已知类名/方法名，需要快速定位
  • 需要分析继承、接口、子类等跨文件关系
  • 需要查看类的公开 API 摘要

✅ 继续用 read/grep 的场景:
  • 读取配置文件、build.gradle、XML 布局
  • 需要深度分析代码逻辑
  • 需要看方法的上下文（前后几十行）
  • 符号未被索引（第三方库、生成代码）
```

## 十一、性能优化手段

### 11.1 索引性能

| 优化 | 效果 |
|------|------|
| 批量事务（200 文件/批） | 减少事务提交次数 |
| WAL 模式 | 提升并发性能 |
| 增量更新 | 只解析变更文件 |
| 文件独立事务 | 单文件失败不影响其他 |

### 11.2 查询性能

| 优化 | 效果 |
|------|------|
| FTS5 全文搜索 | 毫秒级响应 |
| 复合索引 | 覆盖高频查询 |
| 触发器同步 FTS | 自动维护搜索索引 |
